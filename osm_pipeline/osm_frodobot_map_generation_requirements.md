# OSM/FrodoBot Map Image Generation Requirements

## 1. Current Point of Confusion

It seems that the phrase `north-up` caused the intention to be misunderstood.
What I want is **not** simply to fix the map so that north is always at the top.

The key point I want is the following:

> **Generate the map image so that the ego direction, meaning the actual direction the robot is moving, points toward the top of the image.**

So, the “upward direction” here does **not** mean the absolute north direction on the map.
It refers to the robot’s current moving direction or heading direction.

---

## 2. Desired Orientation of the Odometry / OSM Map Image

The odometry map or OSM-based local map image that will be used as model input should be aligned with the robot’s viewpoint.

In other words, the forward-looking direction in the raw camera image and the movement direction in the map image should be consistent with each other.

Therefore, the map image should satisfy the following conditions:

- Generate a local crop centered on the robot’s current position.
- Align the image so that the robot’s heading direction points upward.
- Display the past path as a gray line.
- Display the future path as a red line.
- As the robot moves, generate the map image so that it naturally appears to move forward as well.

In summary, the odometry map / local OSM map should be **ego-heading-up**, not **north-up**.

---

## 3. Difference Between the BEV Map and the Local Odometry Map

The BEV map and the local odometry map serve different purposes.

### 3.1 Local Odometry / OSM Map

The local odometry map is an image that may be used as model input.
Therefore, alignment with the raw image is important.

In this case, the map should be generated so that the robot’s heading direction points upward in the image.

That means the local map should have the following characteristics:

- ego-centric view
- heading-up orientation
- crop centered on the current robot position
- past path: gray
- future path: red

### 3.2 BEV Overview Map

The BEV map is intended as an overview for visually checking the entire segment.
Therefore, unlike the local odometry map, it does not need to rotate according to heading at every frame.

The BEV map only needs to satisfy the following conditions:

- It should show the full trajectory of the segment at a glance.
- It should fit cleanly within a fixed rectangular frame.
- It should not be arbitrarily rotated into a tilted image.
- Very short segments may be shown with a zoomed-in view so that the path is easier to see.
- The start point, end point, and overall flow of the route should be clearly visible.

In other words, the BEV map is not primarily for model input, but for result inspection and visualization.

---

## 4. Direction for Using the OSRM Route vs. the Actual GPS Trajectory

At first, the idea was to use the route generated by OSRM from the start GPS and goal GPS as the fixed path.
However, if the raw camera image and the map image are both used as model inputs, it is more important that the map image is aligned with the actual driving trajectory.

Therefore, the following approach is more reasonable for now:

1. Generate the map image based on the actual GPS trajectory.
2. Create a local crop based on the current position at each frame.
3. Display the trajectory before the current frame in gray.
4. Display the trajectory after the current frame in red.
5. Align the cropped image so that the robot’s heading direction points upward.

This way, the raw image and the odometry/OSM map image can share the same driving flow.

## 6. Final Desired Outcome

Ultimately, what I want is the following:

> Based on the actual GPS trajectory, generate an ego-centric local map image for each frame such that the robot’s heading direction points upward in the image, and use it as a model input aligned with the raw camera image.

In addition, for each segment, a separate BEV overview map should also be generated so that the full driving path and segment quality can be visually checked.

---

## 7. Key Summary

- By “upward direction,” I mean the robot’s ego heading direction, not north.
- The local odometry / OSM map should be generated in an ego-heading-up manner.
- The raw camera image and the map image should share the same movement direction.
- The BEV map is for overview purposes, so it should be generated in a fixed rectangular format without arbitrary rotation.
- Very short segments may be shown with a zoomed-in view in the BEV map.
- First test only Episode 6, visually verify the result, and then expand to the full dataset.
