# OSM FrodoBot 파이프라인: 비타일 방식 지도 접근의 문제점 분석

## 1. 개요

FrodoBots 데이터셋 기반 내비게이션 학습 데이터를 생성하는 과정에서,  
OSM 지도를 **타일 방식(tile-based)** 이 아닌 **비타일 방식(non-tile)** 으로 처리하는 것을 고려했으나  
아래와 같은 이유로 실용적이지 않다는 결론에 도달했다.

---

## 2. Ego-centric 크롭이 어려운 이유

- 타일 방식은 GPS 좌표 → 픽셀 좌표 변환이 Web Mercator 수식으로 **정밀하게 결정**된다.
- 비타일 방식(SVG, Shapefile, 직접 렌더링 등)은 렌더링 해상도, 좌표계, 바운딩 박스 설정에 따라
  픽셀 좌표 매핑이 달라진다.
- Ego-centric 크롭은 **현재 로봇 위치를 정확히 이미지 중심에** 놓아야 하는데,
  비타일 방식에서는 GPS → 픽셀 변환의 오차가 크거나 구현이 복잡해진다.
- 타일 방식은 `latlon_to_pixel_global()` 함수 하나로 임의의 GPS 좌표를
  픽셀 좌표로 변환하고, 캔버스 어디서든 동일한 기준으로 크롭할 수 있다.

---

## 3. GPS 궤적 및 Heading 정렬이 어려운 이유

- 타일 기반 캔버스는 **North-up 방향이 보장**된다 (Web Mercator 표준).
  - 위쪽 = 북쪽, 오른쪽 = 동쪽 → heading 회전 공식이 단순하다.
  - `rot_deg = 90° - heading_deg` 한 줄로 heading-up 정렬 가능.
- 비타일 방식은 렌더링 엔진마다 좌표 방향이 다를 수 있어
  heading 정렬을 위한 추가적인 좌표계 변환이 필요하다.
- GPS 궤적을 지도 위에 올릴 때도 타일 방식은 각 GPS 포인트를
  픽셀 좌표로 즉시 변환하여 `cv2.line()`으로 그릴 수 있으나,
  비타일 방식은 좌표 정규화 과정이 복잡하다.

---

## 4. 고정 직사각형 BEV 생성이 불안정한 이유

- BEV(Bird's Eye View) 이미지는 세그먼트 전체 GPS 경로의
  **bbox를 기준으로 일정한 크기로 crop**해야 한다.
- 타일 방식은 GPS bbox에 해당하는 타일을 다운로드하여 이어붙이면
  자동으로 정확한 영역의 지도 이미지가 만들어진다.
- 비타일 방식(예: Overpass API의 XML 데이터를 직접 렌더링)은:
  - 렌더링 엔진 설치 및 설정이 필요 (Maperitive, Mapnik 등)
  - 렌더링 bbox와 출력 해상도를 수동으로 지정해야 함
  - 세그먼트마다 bbox가 달라 일관된 해상도 보장이 어려움
  - 결과 이미지의 픽셀당 미터 비율(meter-per-pixel)이 일정하지 않음

---

## 5. 텍스트 레이블만 제거하기 어려운 이유

- 현재 사용하는 OSM 타일(`tile.openstreetmap.org`)은
  **서버에서 미리 렌더링된 PNG 래스터 이미지**다.
- 도로, 건물, 텍스트 레이블이 모두 같은 픽셀 레이어에 합쳐져 있어
  후처리로 텍스트만 분리하는 것이 불가능하다.
- 대안으로 시도한 방법들:
  - `tiles.wmflabs.org/osm-no-labels/` → 서버 다운 (응답 없음)
  - `cartodb-basemaps/light_nolabels/` → 도로 정보도 없는 빈 회색 배경만 반환
  - Mapbox/Stamen 스타일 타일 → API 키 및 유료 플랜 필요
  - 로컬 Mapnik/Maperitive 렌더링 → 설치·설정 복잡, 처리 속도 매우 느림
- **결론**: 현재 파이프라인 환경에서 텍스트만 제거하는 것은 현실적으로 불가능하다.
  모델 학습 관점에서 텍스트 레이블은 feature로 학습되지 않으므로 큰 영향이 없다.

---

## 6. 타일 기반 OSM 렌더링이 적합한 이유

| 항목 | 타일 방식 | 비타일 방식 |
|------|-----------|-------------|
| GPS → 픽셀 변환 | Web Mercator 수식으로 정밀 | 렌더러마다 다름 |
| Ego-centric 크롭 | 단순 (중심 좌표 기준) | 복잡 |
| Heading 정렬 | North-up 보장, 회전 공식 단순 | 좌표계 변환 필요 |
| BEV 생성 | bbox 타일 자동 stitch | 해상도·bbox 수동 지정 |
| 캐싱 | 타일 단위 디스크 캐시 | 세그먼트 단위 재렌더링 |
| 처리 속도 | ~200 frames/sec (캐시 적중) | 수 frames/sec |
| 안정성 | tile.openstreetmap.org 안정적 | 엔진 의존성 높음 |

---

## 7. 최종 결론

현재 FrodoBots OSM 내비게이션 파이프라인에서는  
**타일 기반 OSM 렌더링 방식을 유지하는 것이 올바른 선택**이다.

- GPS 좌표와 픽셀 좌표 간의 정밀한 1:1 매핑이 보장된다.
- Ego-centric 크롭, heading-up 정렬, BEV 생성 모두 일관되게 처리할 수 있다.
- 텍스트 레이블 제거는 현재 환경에서 현실적이지 않으며,
  모델이 텍스트를 직접 feature로 사용하지 않으므로 학습에 미치는 영향이 미미하다.
- 타일 캐시를 활용하면 167개 세그먼트 전체를 빠르게 처리할 수 있다.
